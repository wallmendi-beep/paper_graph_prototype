# Design Notes

이 문서는 논문 분석 프로그램의 설계 노트를 누적해서 기록하는 파일이다.

## 운영 원칙

- 이 파일의 기존 내용을 삭제하지 않는다.
- 기존 문장을 직접 고쳐야 할 때도 원문을 지우지 말고, 아래에 `정정 메모` 또는 `후속 메모`로 추가한다.
- 새 아이디어, 변경 결정, 보류 사항은 항상 문서 맨 아래에 날짜와 함께 추가한다.
- 확정된 내용과 가설 단계의 내용을 구분해서 적는다.
- 구현 전에 생긴 생각도 남기되, 나중에 다시 검토할 수 있도록 이유를 함께 적는다.

## 문서 작성 규칙

- 새 항목은 `## YYYY-MM-DD` 형식의 섹션으로 추가한다.
- 각 날짜 섹션 아래에는 `배경`, `핵심 아이디어`, `구조`, `분석 규칙`, `열린 질문`, `다음 단계` 중 필요한 항목만 쓴다.
- 기존 내용을 덮어쓰는 대신 `추가 메모:` 또는 `변경 해석:`으로 이어 쓴다.

---

## 2026-04-21

### 배경

기존 AI 기반 논문 분석은 결과 자체는 훌륭할 수 있지만, 왜 그런 결론이 나왔는지 사용자가 다시 분석해야 하는 문제가 있다.  
즉, 결과는 빠르게 얻을 수 있어도 근거 구조가 충분히 드러나지 않으면 검토 가능성과 신뢰성이 떨어진다.

### 핵심 아이디어

논문 텍스트를 문장과 단어 단위로 분해하고, 각각을 별도 ID를 가진 노드로 관리한다.  
이 노드들을 코드 레벨에서 연결해 그래프를 만들면, 유사도, 중복, 충돌, 고립된 주장, 논리적 비약 후보를 시각적으로 확인할 수 있다.  
이후 AI는 텍스트 전체를 막연히 읽는 대신, 이미 구조화된 그래프를 바탕으로 분석 이유를 설명하고 재점검을 도와주는 보조 계층으로 동작한다.

### 문제 정의

이 프로그램이 풀고자 하는 문제는 "논문을 자동으로 요약하는 것"이 아니다.  
더 정확히는 아래 문제를 다룬다.

- 어떤 문장이 어떤 개념과 연결되는지 사람이 추적하기 어렵다.
- 비슷한 주장과 반복 서술이 문서 전반에 흩어져 있으면 눈으로 찾기 어렵다.
- 상충되는 문장이나 근거가 약한 결론을 빠르게 식별하기 어렵다.
- AI 분석 결과가 나와도, 그 판단 근거를 사용자가 다시 구조적으로 검증하기 어렵다.

### 제품 관점의 차별점

이 도구의 차별점은 AI가 대신 결론을 내려주는 데 있지 않다.  
차별점은 "결론에 이르는 구조를 사람이 볼 수 있게 만든다"는 데 있다.

- 일반 AI 분석기: 결과 중심
- 제안하는 도구: 근거 구조 중심
- 일반 AI 분석기: 블랙박스 해석
- 제안하는 도구: 추적 가능한 그래프 해석
- 일반 AI 분석기: 사람이 다시 원문을 뒤져야 함
- 제안하는 도구: 사람이 그래프와 원문을 함께 보며 재검토 가능

### 핵심 설계 원칙

- AI는 최종 판정자가 아니라 재검토 파트너로 둔다.
- 시스템은 "논리적 허점"을 단정하지 않고 "의심 신호"를 제시한다.
- 원문과 그래프 사이의 왕복 탐색이 쉬워야 한다.
- 분석 결과는 설명 가능해야 하며, 각 판단은 연결 근거를 가져야 한다.
- 한 문장에 대한 평가는 반드시 원문 위치와 연결된 상태여야 한다.

### 기본 데이터 모델

#### 1. 문서 노드

- `document_id`
- 문서 제목 또는 소스 정보
- 전체 문장 수, 토큰 수, 분석 시각 등 메타데이터

#### 2. 문장 노드

- `sentence_id`
- 원문 텍스트
- 문서 내 순서
- 소속 문단 또는 섹션
- polarity, certainty, citation presence 같은 보조 속성

예시:

```text
sentence:42
text: "This method significantly improves retrieval accuracy."
section: Results
polarity: positive
certainty: high
```

#### 3. 개념 토큰 노드

- `token_id`
- 표면형
- 정규화형
- 출현 빈도
- 연결된 문장 목록

#### 4. 링크

- 문서-문장 계층 링크
- 문장-토큰 포함 링크
- 문장-문장 유사도 링크
- 문장-문장 충돌 링크
- 문장-문장 보강/근거 링크

### 분석 계층 설계

이 프로그램은 아래 다섯 층으로 분석을 수행한다.

#### 1. 구조 추출 층

- 문장 분리
- 문단 분리
- 섹션 제목 추출
- 인용 표현 탐지
- 표, 각주, 참고문헌 같은 비본론 영역 분리

#### 2. 개념 추출 층

- 의미 있는 단어 또는 구문 추출
- 표제어 정규화
- 불용어 제거
- 중요 개념의 빈도와 분포 계산

#### 3. 관계 형성 층

- 같은 개념을 공유하는 문장 연결
- 인접 문장 간 논리 흐름 연결
- 섹션을 가로지르는 반복 주장 연결
- 근거 문장과 결론 문장 후보 연결

#### 4. 이상 신호 탐지 층

완전한 자동 판정보다는 아래와 같은 후보 신호를 우선 탐지한다.

- 중복 신호: 거의 같은 의미의 문장이 반복되는 경우
- 충돌 신호: 동일 개념을 공유하면서 polarity 또는 antonym이 충돌하는 경우
- 고립 신호: 다른 문장과 강하게 연결되지 않는 핵심 주장
- 비약 신호: 결론성 문장이 근거 문장과 충분히 연결되지 않는 경우
- 근거 부족 신호: 강한 주장인데 인용, 수치, 근거 표현이 약한 경우

#### 5. AI 협업 층

AI는 다음 역할을 맡는다.

- 그래프에서 표시된 충돌 후보의 해석
- 중복 문장 묶음의 요약
- 고립 문장이 진짜 중요한 독립 주장인지, 분석 누락인지 검토
- 비약 신호가 실제 논리 오류인지 문맥적으로 재판단
- 사람이 선택한 노드 묶음을 자연어 보고서로 변환

### "논리적 허점"에 대한 입장

중요한 점은 시스템이 논리적 허점을 자동으로 단정하면 안 된다는 것이다.  
현재 단계에서 더 현실적인 목표는 "논리적 허점 후보" 또는 "검토 필요 지점"을 표시하는 것이다.

이 입장을 유지하면 다음 장점이 있다.

- 오탐이 있어도 도구 신뢰가 크게 무너지지 않는다.
- 사용자와 AI가 함께 검토하는 구조와 잘 맞는다.
- 규칙 기반 탐지와 LLM 해석을 자연스럽게 결합할 수 있다.

### 추천 사용자 흐름

1. 사용자가 논문 파일을 업로드한다.
2. 시스템이 문장, 토큰, 섹션을 추출한다.
3. 그래프에서 문장-개념 연결과 충돌/중복 후보를 시각화한다.
4. 사용자가 특정 노드나 링크를 클릭해 원문 맥락을 본다.
5. 사용자가 선택한 노드 묶음을 기준으로 AI에게 해석을 요청한다.
6. AI는 그래프 근거를 바탕으로 설명, 반론, 요약, 재검토 포인트를 제시한다.

### MVP 범위 제안

초기 버전에서는 아래까지만 잘 되어도 의미가 크다.

- 문장 노드 생성
- 핵심 토큰 노드 생성
- 문장-토큰 링크 생성
- 문장-문장 유사도 계산
- 충돌 후보 탐지
- 고립 문장 탐지
- 그래프 시각화
- 노드 선택 시 원문과 연결 이유 표시

### 이후 확장 방향

- 문단, 섹션, 주장, 근거를 별도 계층 노드로 확장
- TF-IDF, discourse marker, citation cue 반영
- figure/table caption과 본문 연결
- 참고문헌 인용망 연결
- 논문 간 비교 그래프
- 사용자가 수동으로 링크를 수정하거나 메모를 다는 기능
- AI가 생성한 해석 결과를 다시 그래프 주석으로 저장하는 기능

### 열린 질문

- 한국어와 영어가 섞인 논문에서 토큰 정규화를 어떻게 안정화할 것인가
- 문장 단위만으로 부족할 때 clause 단위 분해가 필요한가
- "근거"와 "결론" 관계를 규칙 기반으로 어디까지 잡을 수 있는가
- 시각화에서 정보량이 많아질 때 어떤 축약 전략을 쓸 것인가
- 사용자에게 보여줄 기본 뷰는 문장 중심이 좋은가, 섹션 중심이 좋은가

### 다음 단계

- 현재 프로토타입의 노드/링크 모델을 설계노트 기준으로 재정렬
- `중복`, `충돌`, `고립`, `비약` 신호를 구분하는 규칙 초안 작성
- UI에서 노드 선택 시 "왜 연결되었는지" 설명하는 패널 강화
- AI 협업 기능을 위한 입력 포맷 초안 정의

### 후속 메모

이 설계의 본질은 "AI가 논문을 대신 읽는 프로그램"이 아니라 "사람과 AI가 함께 검토할 수 있도록 논문의 논리 구조를 외부화하는 프로그램"이라는 점이다.  
따라서 앞으로 기능을 추가할 때도 요약 품질보다 추적 가능성, 설명 가능성, 재검토 가능성을 우선해야 한다.
