## 2026-04-21

### 배경

기존 AI 기반 논문 분석은 결과 자체는 훌륭할 수 있지만, 왜 그런 결론이 나왔는지 사용자가 다시 분석해야 하는 문제가 있다.  
즉, 결과는 빠르게 얻을 수 있어도 근거 구조가 충분히 드러나지 않으면 검토 가능성과 신뢰성이 떨어진다.

### 핵심 아이디어

논문 텍스트를 문장과 단어 단위로 분해하고, 각각을 별도 ID를 가진 노드로 관리한다.  
이 노드들을 코드 레벨에서 연결해 그래프를 만들면, 유사도, 중복, 충돌, 고립된 주장, 논리적 비약 후보를 시각적으로 확인할 수 있다.  
이후 AI는 텍스트 전체를 막연히 읽는 대신, 이미 구조화된 그래프를 바탕으로 분석 이유를 설명하고 재점검을 도와주는 보조 계층으로 동작한다.

### 문제 정의

이 프로그램이 풀고자 하는 문제는 "논문을 자동으로 요약하는 것"이 아니다.  
더 정확히는 아래 문제를 다룬다.

- 어떤 문장이 어떤 개념과 연결되는지 사람이 추적하기 어렵다.
- 비슷한 주장과 반복 서술이 문서 전반에 흩어져 있으면 눈으로 찾기 어렵다.
- 상충되는 문장이나 근거가 약한 결론을 빠르게 식별하기 어렵다.
- AI 분석 결과가 나와도, 그 판단 근거를 사용자가 다시 구조적으로 검증하기 어렵다.

### 제품 관점의 차별점

이 도구의 차별점은 AI가 대신 결론을 내려주는 데 있지 않다.  
차별점은 "결론에 이르는 구조를 사람이 볼 수 있게 만든다"는 데 있다.

- 일반 AI 분석기: 결과 중심
- 제안하는 도구: 근거 구조 중심
- 일반 AI 분석기: 블랙박스 해석
- 제안하는 도구: 추적 가능한 그래프 해석
- 일반 AI 분석기: 사람이 다시 원문을 뒤져야 함
- 제안하는 도구: 사람이 그래프와 원문을 함께 보며 재검토 가능

### 핵심 설계 원칙

- AI는 최종 판정자가 아니라 재검토 파트너로 둔다.
- 시스템은 "논리적 허점"을 단정하지 않고 "의심 신호"를 제시한다.
- 원문과 그래프 사이의 왕복 탐색이 쉬워야 한다.
- 분석 결과는 설명 가능해야 하며, 각 판단은 연결 근거를 가져야 한다.
- 한 문장에 대한 평가는 반드시 원문 위치와 연결된 상태여야 한다.

### 기본 데이터 모델

#### 1. 문서 노드

- `document_id`
- 문서 제목 또는 소스 정보
- 전체 문장 수, 토큰 수, 분석 시각 등 메타데이터

#### 2. 문장 노드

- `sentence_id`
- 원문 텍스트
- 문서 내 순서
- 소속 문단 또는 섹션
- polarity, certainty, citation presence 같은 보조 속성

예시:

```text
sentence:42
text: "This method significantly improves retrieval accuracy."
section: Results
polarity: positive
certainty: high
```

#### 3. 개념 토큰 노드

- `token_id`
- 표면형
- 정규화형
- 출현 빈도
- 연결된 문장 목록

#### 4. 링크

- 문서-문장 계층 링크
- 문장-토큰 포함 링크
- 문장-문장 유사도 링크
- 문장-문장 충돌 링크
- 문장-문장 보강/근거 링크

### 분석 계층 설계

이 프로그램은 아래 다섯 층으로 분석을 수행한다.

#### 1. 구조 추출 층

- 문장 분리
- 문단 분리
- 섹션 제목 추출
- 인용 표현 탐지
- 표, 각주, 참고문헌 같은 비본론 영역 분리

#### 2. 개념 추출 층

- 의미 있는 단어 또는 구문 추출
- 표제어 정규화
- 불용어 제거
- 중요 개념의 빈도와 분포 계산

#### 3. 관계 형성 층

- 같은 개념을 공유하는 문장 연결
- 인접 문장 간 논리 흐름 연결
- 섹션을 가로지르는 반복 주장 연결
- 근거 문장과 결론 문장 후보 연결

#### 4. 이상 신호 탐지 층

완전한 자동 판정보다는 아래와 같은 후보 신호를 우선 탐지한다.

- 중복 신호: 거의 같은 의미의 문장이 반복되는 경우
- 충돌 신호: 동일 개념을 공유하면서 polarity 또는 antonym이 충돌하는 경우
- 고립 신호: 다른 문장과 강하게 연결되지 않는 핵심 주장
- 비약 신호: 결론성 문장이 근거 문장과 충분히 연결되지 않는 경우
- 근거 부족 신호: 강한 주장인데 인용, 수치, 근거 표현이 약한 경우

#### 5. AI 협업 층

AI는 다음 역할을 맡는다.

- 그래프에서 표시된 충돌 후보의 해석
- 중복 문장 묶음의 요약
- 고립 문장이 진짜 중요한 독립 주장인지, 분석 누락인지 검토
- 비약 신호가 실제 논리 오류인지 문맥적으로 재판단
- 사람이 선택한 노드 묶음을 자연어 보고서로 변환

### "논리적 허점"에 대한 입장

중요한 점은 시스템이 논리적 허점을 자동으로 단정하면 안 된다는 것이다.  
현재 단계에서 더 현실적인 목표는 "논리적 허점 후보" 또는 "검토 필요 지점"을 표시하는 것이다.

이 입장을 유지하면 다음 장점이 있다.

- 오탐이 있어도 도구 신뢰가 크게 무너지지 않는다.
- 사용자와 AI가 함께 검토하는 구조와 잘 맞는다.
- 규칙 기반 탐지와 LLM 해석을 자연스럽게 결합할 수 있다.

### 추천 사용자 흐름

1. 사용자가 논문 파일을 업로드한다.
2. 시스템이 문장, 토큰, 섹션을 추출한다.
3. 그래프에서 문장-개념 연결과 충돌/중복 후보를 시각화한다.
4. 사용자가 특정 노드나 링크를 클릭해 원문 맥락을 본다.
5. 사용자가 선택한 노드 묶음을 기준으로 AI에게 해석을 요청한다.
6. AI는 그래프 근거를 바탕으로 설명, 반론, 요약, 재검토 포인트를 제시한다.

### MVP 범위 제안

초기 버전에서는 아래까지만 잘 되어도 의미가 크다.

- 문장 노드 생성
- 핵심 토큰 노드 생성
- 문장-토큰 링크 생성
- 문장-문장 유사도 계산
- 충돌 후보 탐지
- 고립 문장 탐지
- 그래프 시각화
- 노드 선택 시 원문과 연결 이유 표시

### 이후 확장 방향

- 문단, 섹션, 주장, 근거를 별도 계층 노드로 확장
- TF-IDF, discourse marker, citation cue 반영
- figure/table caption과 본문 연결
- 참고문헌 인용망 연결
- 논문 간 비교 그래프
- 사용자가 수동으로 링크를 수정하거나 메모를 다는 기능
- AI가 생성한 해석 결과를 다시 그래프 주석으로 저장하는 기능

### 열린 질문

- 한국어와 영어가 섞인 논문에서 토큰 정규화를 어떻게 안정화할 것인가
- 문장 단위만으로 부족할 때 clause 단위 분해가 필요한가
- "근거"와 "결론" 관계를 규칙 기반으로 어디까지 잡을 수 있는가
- 시각화에서 정보량이 많아질 때 어떤 축약 전략을 쓸 것인가
- 사용자에게 보여줄 기본 뷰는 문장 중심이 좋은가, 섹션 중심이 좋은가

### 다음 단계

- 현재 프로토타입의 노드/링크 모델을 설계노트 기준으로 재정렬
- `중복`, `충돌`, `고립`, `비약` 신호를 구분하는 규칙 초안 작성
- UI에서 노드 선택 시 "왜 연결되었는지" 설명하는 패널 강화
- AI 협업 기능을 위한 입력 포맷 초안 정의

### 후속 메모

이 설계의 본질은 "AI가 논문을 대신 읽는 프로그램"이 아니라 "사람과 AI가 함께 검토할 수 있도록 논문의 논리 구조를 외부화하는 프로그램"이라는 점이다.  
따라서 앞으로 기능을 추가할 때도 요약 품질보다 추적 가능성, 설명 가능성, 재검토 가능성을 우선해야 한다.

### 관찰

해당 저장소들은 공통적으로 일반 사용자용 문서 분석 앱이라기보다, argument mining 문제를 다루는 연구 코드에 가깝다.

- 입력이 논문 파일 자체라기보다 특정 topic 또는 데이터셋 중심이다.
- 실행 흐름이 업로드-탐색-시각화가 아니라 데이터 준비-학습-예측 파이프라인 중심이다.
- 결과물도 인터랙티브 그래프 뷰보다 분류기 결과 또는 relation prediction에 가깝다.
- 외부 데이터, 임베딩, 구버전 Python 환경 등 재현 비용이 높은 편이다.

### 우리 프로젝트와 겹치는 부분

- 텍스트를 논리 관계 단위로 해석하려는 시도
- claim, premise, support, attack 같은 관계 개념
- 구조화된 중간 표현을 만든 뒤 관계를 분석하는 접근
- 문장 간 관계를 그래프 관점으로 보는 발상

### 우리 프로젝트와 다른 부분

- 우리는 논문이나 문서를 바로 넣고 탐색하는 도구를 지향한다.
- 우리는 문장-개념 노드를 함께 시각화하고, 유사도, 중복, 충돌, 고립, 비약 후보를 함께 본다.
- 우리는 최종 분류 정확도보다 설명 가능성과 재검토 가능성을 우선한다.
- 우리는 AI를 분류기 자체보다 그래프 해석 보조자로 두려 한다.

### 참고할 것

- `evidence extraction` 개념
- `claim/premise/support/attack` 관계 체계
- 구조 추출 후 관계 판단으로 넘어가는 파이프라인 설계
- 특정 논점 중심으로 문장을 묶어 보는 관점

### 당장 가져오지 않을 것

- 다중 Python 버전 의존 환경
- 메일 기반 외부 데이터 다운로드 구조
- 학습 중심 실행 흐름
- 무거운 지식그래프, 임베딩 의존성
- 데이터셋 성능 중심의 설계 우선순위

### 판단

이 계열 저장소는 우리와 완전히 같은 제품은 아니다.  
다만 "논리 관계를 어떤 타입으로 모델링할 것인가"를 고민할 때 참고할 수 있는 선행 연구 축으로는 가치가 있다.

### 변경 해석

현재 프로젝트는 `argument mining app`으로 좁게 정의하기보다, 아래의 조합으로 정의하는 편이 더 정확하다.

- 문장 유사도 분석
- 개념 토큰 그래프화
- 충돌/중복/고립/비약 후보 탐지
- 시각적 재검토
- AI 보조 해석

즉, argument mining은 전체 시스템의 일부 참고축이지, 제품 전체를 설명하는 단일 키워드는 아니다.

### 1. 참고할 구조

`claim-mapper`에서 가장 먼저 참고할 부분은 특정 도메인보다 그래프를 다루는 방식 자체이다.

관찰된 좋은 점:

- 노드 타입과 링크 타입을 공통 스키마로 분리한다.
- 분석 결과를 바로 렌더링하지 않고, 그래프 데이터로 변환하는 중간 계층이 있다.
- 그래프 필터링, 선택, 확대/축소, 고립 노드 숨김 같은 검토 UX가 기본 설계에 들어 있다.
- 링크가 단순 연결선이 아니라 `supports`, `contradicts`, `concludes` 같은 의미를 가진다.

### 2. 우리 프로젝트에 번역해서 가져올 것

#### 노드 타입 설계

`claim-mapper`의 `claim/evidence/reasoning` 구조를 그대로 쓰기보다, 우리 도메인에 맞게 아래처럼 확장 해석한다.

- `document`
- `sentence`
- `concept`
- `issue`

후속 확장 후보:

- `claim-candidate`
- `evidence-candidate`
- `section`

#### 링크 타입 설계

우리 프로젝트에서 기본 링크 타입은 다음과 같이 정리할 수 있다.

- `contains`
- `mentions`
- `similar`
- `contradicts`
- `supports`
- `leaps`

후속 확장 후보:

- `repeats`
- `cites`
- `explains`

### 3. 가장 중요한 참고점: 변환 레이어

`claim-mapper`에서 특히 중요한 점은 원본 분석 결과와 시각화용 그래프 모델 사이에 변환 계층이 있다는 것이다.

우리 쪽에서도 다음 레이어를 별도로 두는 것이 바람직하다.

- 입력: 파싱 결과, 유사도 분석 결과, 충돌 후보, 고립 문장, 비약 후보
- 출력: 화면에서 바로 렌더링할 수 있는 `graphData`

예상 함수 형태:

```text
paperAnalysisToGraph(parsed, analysis, signals) -> graphData
```

이렇게 분리하면 얻는 장점:

- 분석 규칙과 시각화가 분리된다.
- D3 구현을 바꿔도 분석 코어를 건드리지 않아도 된다.
- 나중에 AI가 읽는 입력 포맷과 UI 그래프 포맷을 정리하기 쉬워진다.

### 4. UX 측면에서 가져올 것

현재 프로젝트에서도 아래 UX는 우선순위가 높다.

- 노드 타입별 색상 구분
- 링크 타입별 색상/스타일 구분
- 선택 노드 강조
- 필터 토글
- 고립 노드 숨기기 옵션
- 줌, 팬, 드래그
- 현재 표시 중인 노드 수/링크 수 표시
- 빈 상태와 오류 상태 메시지 제공

### 5. 특히 좋은 관점

`claim-mapper`는 그래프를 단순한 시각 효과가 아니라 "검토 인터페이스"로 다루고 있다는 점이 중요하다.

이 관점은 우리 프로젝트에도 그대로 유효하다.

- 사용자는 그래프를 본다.
- 특정 노드 또는 링크를 선택한다.
- 시스템은 왜 연결되었는지 설명한다.
- 사용자는 그 결과를 다시 AI와 함께 검토한다.

즉, 그래프는 결과 화면이 아니라 검토 도구여야 한다.

### 6. 당장 가져오지 않을 것

- 백엔드 전체 구조
- ML 서비스 전체 구조
- 협업 기능
- 검색 기능
- 배포/인증 관련 설정

현재 단계에서는 과도한 범위이며, 핵심 가설 검증에 필요하지 않다.

### 7. 현재 프로젝트에 대한 해석 변화

이번 검토를 통해 현재 프로토타입은 "분석 결과를 보여주는 그래프" 수준에서 더 나아가, "검토를 수행하게 만드는 그래프 인터페이스"로 진화해야 한다는 점이 분명해졌다.

향후 구현 우선순위는 다음 질문에 맞춰 잡는 것이 좋다.

- 이 노드는 왜 존재하는가
- 이 링크는 왜 생겼는가
- 이 연결은 어떤 규칙에 의해 만들어졌는가
- 사용자는 이 지점에서 무엇을 다시 판단해야 하는가

### 1. 전체 판단

이 저장소는 우리 프로젝트와 도메인은 다르지만, 그래프를 "탐색형 검토 인터페이스"로 구성하는 방식이 매우 참고할 만하다.

강점:

- 업로드 이후 바로 탐색 가능한 단일 화면 구조
- 검색, 필터, 통계, 그래프, 상세 패널이 한 흐름에 연결됨
- 노드를 선택하면서 이동하는 탐색 경험이 잘 드러남
- D3 기반 그래프에 검토 중심 UX가 덧붙어 있음

### 2. 특히 참고할 UX 요소

#### 검색 하이라이트

검색 결과를 단순히 필터링해서 숨기는 데 그치지 않고, 노드와 링크를 시각적으로 강조한다.  
이 방식은 우리 프로젝트에서도 유용하다.

우리 쪽 적용 예:

- 문장 본문 검색
- 개념 토큰 검색
- 충돌 이유 검색
- 특정 섹션명 검색

#### 상세 패널

선택한 노드에 대해 오른쪽 패널에서 아래 정보를 보여주는 구조가 좋다.

- 노드 제목
- 노드 타입
- 관찰값 또는 메모
- inbound 관계
- outbound 관계
- 연결 노드로 바로 이동하는 버튼

우리 쪽 적용 예:

- 선택 문장 원문
- 연결된 개념 목록
- 유사 문장 목록
- 충돌 후보 목록
- 선택 노드가 왜 중요하게 잡혔는지 설명

#### 탐색 히스토리

노드를 따라 이동한 기록을 뒤로/앞으로 탐색할 수 있는 구조가 있다.  
이건 사용자가 복잡한 그래프를 검토할 때 매우 중요하다.

우리 쪽 적용 예:

- 문장 A -> 개념 X -> 문장 B -> 충돌 후보 문장 C 식으로 이동한 흐름 저장
- 이전 노드로 빠르게 돌아가기
- 비교 중이던 위치를 잃지 않기

#### 그래프 상태 가시화

현재 보이는 엔티티 수, 관계 수, 타입 수를 뱃지 형태로 보여주는 점이 좋다.

우리 쪽 적용 예:

- 문장 수
- 개념 수
- 강한 연결 수
- 충돌 후보 수
- 고립 문장 수

#### 명시적 뷰 컨트롤

마우스 휠에만 의존하지 않고 확대, 축소, 맞춤, 리셋 버튼을 제공하는 것이 좋다.

우리 쪽 적용 예:

- `Fit to screen`
- `Reset view`
- `충돌만 보기`
- `고립 문장 숨기기`

### 3. 우리 프로젝트에 주는 해석

이 검토를 통해 우리 앱의 그래프는 "결과 그림"이 아니라 아래 세 가지 역할을 동시에 해야 한다는 점이 분명해졌다.

- 탐색 인터페이스
- 검토 인터페이스
- AI 재질문을 위한 선택 인터페이스

즉 사용자는 그래프에서 단순히 보고 끝나는 것이 아니라, 선택하고, 따라가고, 비교하고, 다시 질문할 수 있어야 한다.

### 4. 당장 가져오지 않을 것

- `memory.json` 전용 입력 형식
- 도메인 특화 엔티티 타입
- 다크 모드
- 내보내기 기능
- 키보드 단축키 도움말 오버레이

현재 단계에서는 핵심 가설 검증과 직접 관련이 적다.

- 선택 노드 상세 패널 강화
- 연결 노드 클릭 이동
- 검색 하이라이트
- 그래프 상태 뱃지
